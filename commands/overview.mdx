signal-bot-orx responds to messages in two ways depending on context: in group chats it only replies when you mention it by name, and in direct messages every message triggers a reply. Slash commands like `/search`, `/imagine`, and `/weather` work the same way in both contexts — you invoke them explicitly, and the bot runs the corresponding action.

## Chat triggers

### Group chats

In a group chat, the bot only processes a message when you mention it. By default, the bot responds to `@signalbot` and `@bot`. You can add custom aliases by setting the `BOT_MENTION_ALIASES` environment variable to a comma-separated list:

```bash
BOT_MENTION_ALIASES=@signalbot,@bot,@mybot
```

The bot detects mentions both from Signal's metadata mention system and from plain-text alias matching as a fallback. If the incoming message includes a valid mention in the Signal metadata, that takes priority.

### Direct messages

In a DM, every incoming message triggers a reply — no mention prefix needed. The bot reads your message as the full prompt.

### Telegram

On Telegram, the bot responds in group chats when you mention it by `@bot_username` or when you reply to one of the bot's own messages. Set `TELEGRAM_BOT_USERNAME` in your environment to enable accurate username matching. In Telegram DMs, all messages trigger a reply.

<Note>
  Slash commands (`/search`, `/imagine`, `/weather`, etc.) bypass the mention check. The bot processes them regardless of whether you included a mention, as long as the message comes from an authorized sender.
</Note>

## Prompt limits

The bot enforces a maximum prompt length controlled by `BOT_MAX_PROMPT_CHARS` (default: `700`). This applies to both chat messages and slash command queries.

If your message exceeds the limit, the bot replies:

```text
Prompt too long. Maximum is 700 characters.
```

To raise the limit, set `BOT_MAX_PROMPT_CHARS` in your environment to a higher integer value.

## Webhook endpoints

The bot exposes the following HTTP endpoints:

| Endpoint | Method | Description |
|---|---|---|
| `/healthz` | `GET` | Health check. Returns `{"status": "ok"}` when the server is running. |
| `/webhook/signal` | `POST` | Signal transport endpoint. Receives incoming messages from `signal-cli-rest-api`. |
| `/webhook/whatsapp` | `POST` | WhatsApp bridge endpoint. Only active when `WHATSAPP_ENABLED=true`. |
| `/webhook/telegram` | `POST` | Telegram webhook endpoint. Only active when `TELEGRAM_ENABLED=true`. |

<Info>
  The WhatsApp and Telegram endpoints return `{"status": "ignored", "reason": "whatsapp_disabled"}` (or `telegram_disabled`) if the respective transport is not enabled, so you do not need to restrict routing at the network level.
</Info>
